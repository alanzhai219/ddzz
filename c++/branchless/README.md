# Branchless Cases

这个目录用 9 个小案例说明一件事：`branchless` 不是“去掉 `if`”这么简单，它的核心目标是降低不可预测分支带来的流水线清空、前端停顿和投机失败成本。

对应实现文件：

- `case1_hex.hpp`: 十六进制解码
- `case2_filter_positive.hpp`: 过滤负数
- `case3_unrolled_sum.hpp`: 循环展开
- `case4_compare_date.hpp`: 日期比较
- `case5_clamp.hpp`: clamp
- `case6_abs.hpp`: 绝对值
- `case7_mask_select.hpp`: 掩码选择
- `case8_sentinel_search.hpp`: sentinel 查找
- `case9_bit_scan.hpp`: 位图扫描
- `test_branchless.cpp`: 正确性校验 + benchmark + perf branch/miss 统计

## 背景

现代 CPU 遇到条件分支时，会先依赖分支预测器猜测下一条执行路径。猜对时成本很低，猜错时通常要回滚投机执行结果，并重新灌满流水线。对于数据分布随机、分支方向难预测的代码，这个代价会非常明显。

`branchless` 的常见思路不是让程序“没有控制流”，而是把原本依赖分支选择的逻辑，改成下面几类形式：

- 用查表把多路判断变成一次索引
- 用布尔值参与算术，代替条件跳转
- 用条件移动或位运算表达 `min/max/clamp/abs`
- 用数据编码简化比较路径
- 用循环展开降低分支频率

但 branchless 也有代价：

- 可能引入更多内存访问
- 可能增加无效写入或额外算术
- 可能让代码更难读
- 如果原始分支本来就高度可预测，branchless 版本不一定更快

所以这个目录的重点不是“branchless 必赢”，而是观察不同模式下 `branches`、`branch-misses` 和总耗时之间的关系。

## Case 1: 查表法解码十六进制

文件：`case1_hex.hpp`

有分支版本按字符范围逐段判断：

- `'0' ~ '9'`
- `'A' ~ 'F'`
- `'a' ~ 'f'`

这种写法的问题是，同一次解码需要经历多次条件判断，而且输入字符分布如果比较随机，预测器不容易稳定收敛。

branchless 版本把字符直接映射到一个 `256` 项查找表：

- 字符值直接作为索引
- 非法字符预填成 `-1`
- 合法字符预填成对应数值

设计原理：

- 把“控制选择”改成“数据选择”
- 用一次内存读代替多次条件跳转
- 避免同一输入走不同控制路径

适用前提：

- 输入域有限且离散
- 可以接受一张常驻缓存的小表

典型代价：

- 增加一次表访问
- 如果表很大、访问模式差，可能受缓存影响

## Case 2: 条件写入 vs 推测性写入

文件：`case2_filter_positive.hpp`

目标是把数组里的非负数压缩到前面。

有分支版本：

- 先判断 `values[i] >= 0`
- 成立时才写入 `values[pos++]`

如果正负数分布接近各半，这个分支就非常难预测。

branchless 版本的思路是：

- 不管条件是否成立，先执行 `values[pos] = values[i]`
- 再用布尔值控制 `pos` 是否递增

位运算版本进一步把布尔判断写成显式位操作。

设计原理：

- 保留相同的数据结果
- 牺牲一部分“多余写入”，换取更稳定的执行路径
- 让 CPU 不需要根据数据分布来猜测是否跳转

这个案例很典型，因为它体现了 branchless 的一个常见交换：

- 少分支
- 多执行一些原本可以跳过的操作

适用前提：

- 额外写入成本比 mispredict 成本更低
- 写入目标不会引入额外语义问题

风险：

- 写放大
- 对缓存和 store buffer 的压力可能上升

## Case 3: 循环展开

文件：`case3_unrolled_sum.hpp`

这里不是真正“把条件改成算术”，而是通过展开循环减少分支出现频率。

简单循环每处理一个元素就要经历一次回边分支。即使这个循环分支通常高度可预测，它仍然会占用前端带宽和分支资源。

展开后的版本一次处理 4 个元素：

- 回边分支次数减少到原来的约 `1/4`
- 指令级并行机会更多
- 循环控制逻辑的占比下降

设计原理：

- 不是消灭不可预测分支
- 而是减少“分支密度”
- 让真正有用的计算占更多比例

适用前提：

- 循环体足够简单
- 展开后不会造成明显的指令缓存压力

风险：

- 代码体积变大
- 展开过度可能反而伤害 I-cache

## Case 4: 编码后比较日期

文件：`case4_compare_date.hpp`

有分支版本按 `year -> month -> day` 逐字段比较：

- 年不同就返回
- 年相同再看月
- 月相同再看日

这是非常自然的写法，但控制路径取决于数据分布。如果年份经常相同、月份随机，分支行为会变得更复杂。

branchless 版本先把日期编码成 `YYYYMMDD` 整数，再做一次比较。

设计原理：

- 把多层级比较收缩成单一标量比较
- 让字段级分支消失
- 依赖数值编码保持字典序

这个案例很适合说明一个重要事实：

- `branch-misses` 降低，不代表整体一定更快

因为编码日期本身引入了额外乘法和加法。如果原始分支已经足够可预测，这些额外算术成本可能会盖过分支收益。

适用前提：

- 可以构造保持排序语义的编码
- 编码成本低于多级分支成本

## Case 5: clamp 的算术化

文件：`case5_clamp.hpp`

目标是把 `val` 限制在 `[lo, hi]` 区间。

有分支版本非常直接：

- 小于下界就返回下界
- 大于上界就返回上界
- 否则返回原值

branchless 版本有两条路线：

- 用三元表达式表达 `max`/`min`，依赖编译器生成 `cmov`
- 用位运算手工实现 `branchless_max` 和 `branchless_min`

设计原理：

- 让“选择哪个值”变成寄存器间的数据流选择
- 尽量避免控制流跳转

这个案例也说明，branchless 不是只看源码有没有 `if`。有些三元表达式在汇编层面会落成条件移动，不一定形成真正分支；也有些位运算版本虽然“更像 branchless”，但指令更多、依赖链更长，未必更快。

适用前提：

- 平台和编译器能把这种表达式优化成有利形式
- 额外算术不会拉长关键路径太多

## Case 6: 绝对值的位技巧

文件：`case6_abs.hpp`

有分支版本：

- 如果小于 0，返回 `-v`
- 否则返回 `v`

branchless 版本利用补码性质：

- `mask = v >> 31`
- 正数时 `mask = 0`
- 负数时 `mask = -1`
- `(v ^ mask) - mask` 就会得到绝对值

设计原理：

- 把符号判断折叠进位运算
- 避免显式条件跳转

这个例子本身非常短，也正因为如此，经常能观察到一个现象：即使 branchless 版本 branch miss 更少，整体时间也未必显著下降。原因是原始版本已经很短，编译器也可能自行把它优化成无分支或近似无分支的形式。

## Case 7: 掩码选择

文件：`case7_mask_select.hpp`

这个案例把“条件二选一”写成掩码混合。

有分支版本：

- `cond` 成立返回 `when_true`
- 否则返回 `when_false`

branchless 版本先把条件扩成全 0 / 全 1 掩码，再做按位选择。

设计原理：

- 把控制流里的分叉变成数据通路里的 blend
- 适合简单标量选择和固定宽度整数处理

注意点：

- 这种写法可读性通常不如三元表达式
- 对于简单选择，编译器也可能把三元直接优化成 `cmov`

## Case 8: Sentinel 查找

文件：`case8_sentinel_search.hpp`

这个案例不是纯按位 branchless，而是典型的“把循环内边界判断移出热路径”。

有分支版本每轮都做两件事：

- 检查 `i < length`
- 检查 `data[i] == target`

sentinel 版本先把 `target` 放到 `data[length]`，再让循环只保留“是否命中目标”的判断，最后在循环外确认是否真的在有效范围内命中。

设计原理：

- 用额外一个哨兵槽位，换掉循环里的边界分支
- 减少热路径里的控制逻辑密度

适用前提：

- 可以安全写入一个额外 sentinel 位置
- 数据结构允许临时修改尾部元素

## Case 9: 位图扫描 vs `ctz`

文件：`case9_bit_scan.hpp`

目标是在一个 32 位 bitmask 里找出最低位的 `1` 在哪里。

有分支版本逐位扫描：

- 从 bit 0 开始
- 逐位测试是否命中

branchless 风格版本把这个过程交给位扫描指令语义：

- `__builtin_ctz(mask)` 直接返回最低位 `1` 前面的 0 的个数

设计原理：

- 把线性多分支搜索压缩成单条位操作指令或极短序列
- 非常适合 bitmap allocator、稀疏索引、token 扫描一类场景

注意点：

- `ctz(0)` 是未定义的，所以需要额外约束输入非零，或者单独处理零值
- 这一类 trick 通常强依赖 ISA 和编译器内建函数

## 更多 Branchless Trick

除了这 9 个案例，常见的 branchless / 去分支技巧还有：

- 布尔比较直接参与算术，例如 `pos += (x > 0)`
- SIMD compare + blend，把标量条件选择扩展到向量路径
- halo padding，在数组四周补边界，消掉内层越界判断
- 查表驱动状态机，把 parser 的状态跳转变成状态表索引
- 固定流程网络，例如 sorting network，用固定比较交换序列替代数据相关分支
- 双阶段处理，先生成 flags / mask，再统一压缩或 scatter

这些方法的共同目标都一样：尽量让热路径更接近稳定的数据流，而不是让执行路径跟着输入频繁分叉。

## 如何解读 benchmark

`test_branchless.cpp` 用 Linux `perf_event_open` 统计两类硬件事件：

- `PERF_COUNT_HW_BRANCH_INSTRUCTIONS`
- `PERF_COUNT_HW_BRANCH_MISSES`

建议同时看 3 个指标：

- 总时间
- branches 总数
- miss rate

不要只看其中一个。

常见情况有三种：

1. `branch-misses` 明显下降，时间也下降：branchless 改写有效。
2. `branch-misses` 下降，但时间没变甚至更慢：说明额外算术、访存或写入抵消了收益。
3. `branch-misses` 本来就很低：说明原始分支已经可预测，branchless 可能没有太大价值。

## 结论

这 9 个案例对应了几类常见的 branchless 设计模式：

- 查表
- 推测性写入
- 循环展开
- 编码比较
- 条件移动
- 位运算技巧
- 掩码选择
- sentinel
- 位扫描

它们的共同点是：把“依赖控制流做选择”改成“依赖数据流做选择”。

但最终是否值得使用，必须回到真实数据分布、编译器生成代码和硬件计数结果上判断，而不是只看源码里有没有 `if`。
